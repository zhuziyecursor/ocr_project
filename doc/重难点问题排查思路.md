# 重难点问题排查思路分析

> 本文档记录开发过程中遇到的重难点问题，重点梳理排查思路，供团队参考。

---

## 问题一：PDF 渲染失败 —— Vue 3 响应式系统与 pdfjs 私有字段冲突

**日期：** 2026-04-16  
**耗时：** 同事排查一上午未解决  
**涉及文件：** `frontend/src/components/compare/PDFViewer.vue`

---

### 一、问题现象

- 访问 `http://localhost:3000/pdf-test.html` → PDF 正常渲染
- 访问前端核验页面，左侧源文件区域 → 渲染失败，报错：

```
PDF加载失败: Cannot read private member #pagePromises from an object whose class did not declare it
```

---

### 二、排查思路

#### 第一步：找到"能跑"与"不能跑"的边界

两个场景唯一的差异：

| 场景 | 运行环境 |
|------|---------|
| `pdf-test.html` 正常 | 浏览器原生 `<script type="module">`，不经过 Vite |
| 前端页面失败 | Vue 组件，经过 Vite 打包 + Vue 响应式系统 |

**结论：问题出在 Vite 或 Vue 这一层，与 pdfjs 本身无关。**

#### 第二步：精读报错信息

```
Cannot read private member #pagePromises from an object whose class did not declare it
```

这句话的精确含义：**访问 `#pagePromises` 时，`this` 不是声明这个私有字段的那个类的实例。**

JS 私有字段的规则：只有通过**原始类实例**才能访问，任何包装（Proxy、跨模块实例）都会触发此错误。

由此得出两个排查方向：
1. **多实例**：`getDocument()` 和 `getPage()` 来自两个不同的 pdfjs 模块副本
2. **对象被包装**：pdfjs 返回的对象被某个东西套了一层，`this` 变了

#### 第三步：用堆栈信息排除多实例

查看完整报错堆栈，两处 `getPage` 都指向同一个 `/pdf.mjs` 文件的同一行，说明**不存在多实例**，方向1排除。

#### 第四步：锁定"谁包装了对象"

排除多实例后，唯一可能就是**对象在 Vue 内部被包装**。定位到代码：

```js
const pdfDoc = ref(null)  // ← 嫌疑点
```

Vue 3 的 `ref()` 在存入对象时会调用内部的 `reactive()`，对对象做**深度 Proxy 包装**。此后 `pdfDoc.value` 拿到的不是原始的 `PDFDocumentProxy` 实例，而是它的 Proxy 代理。

当 pdfjs 内部执行 `this.#pagePromises` 时，`this` 是 Proxy，不是原始类实例，JS 引擎拒绝访问私有字段，抛出上述错误。

#### 第五步：验证假设，实施修复

假设成立，解决方案明确：**阻止 Vue 对 pdfjs 对象做深度代理。**

Vue 提供了两种工具：
- `shallowRef()`：只追踪 `.value` 引用本身，不深度代理内部属性
- `markRaw()`：标记对象永远不被 Vue 代理

**修复方式：**

```js
// ❌ 修复前 —— ref() 会深度代理，破坏私有字段访问
const pdfDoc = ref(null)
const pages = ref([])

// ✅ 修复后 —— shallowRef() 只追踪引用，不代理内部对象
const pdfDoc = shallowRef(null)
const pages = shallowRef([])
```

**附带注意事项：** `shallowRef` 包裹的数组，`.push()` 不会触发视图更新，必须重新赋值：

```js
// ❌ 不触发更新
pages.value.push({ num: i, page })

// ✅ 正确写法
pages.value = [...pages.value, { num: i, page }]
```

---

### 三、为什么容易走弯路

同事排查方向集中在：pdfjs 版本兼容性、Vite 配置（optimizeDeps）、worker 加载路径等，这些都是**表面现象**，本质上没有抓住"是谁改变了 `this`"这个核心。

常见无效方案：

| 尝试方案 | 为何无效 |
|---------|---------|
| 改用 `legacy/build/pdf.mjs` | legacy 是 CJS 格式，Vite 处理时问题依旧 |
| 调整 `optimizeDeps.include/exclude` | 不是 Vite 预构建问题 |
| 用 `import.meta.url` 指定 workerSrc | 不是 worker 多实例问题 |
| 用动态 `import()` 加载 public 目录 | Vite 明确禁止 import public 目录的 JS 文件 |

---

### 四、通用排查原则（可复用）

1. **先找边界**：找到"能跑"和"不能跑"的最小差异，问题范围立即缩小一半。
2. **相信报错信息**：报错说什么就是什么，不要绕开它，要正面解释它。本例中"class did not declare it"直接指向 `this` 被替换，顺着这条线找到 Vue Proxy 即可。
3. **第三方库对象存入 Vue 响应式时，优先用 `shallowRef` / `markRaw`**：凡是含有私有字段、内部状态、原生 API 对象（canvas、WebSocket、pdfjs、mapbox 等）都不应该被 Vue 深度代理。

## 问题二：核验页面右侧 OCR 结果渲染空白

**日期：** 2026-04-17
**涉及文件：** `backend/ocr/wrapper.py`、`backend/tasks/document_tasks.py`、`frontend/src/views/Review.vue`

---

### 一、问题现象

- 核验页面左侧 PDF 原文正常
- 右侧 OCR 识别结果区域完全空白，没有任何文字渲染
- `http://localhost:3001/ocr-local-test.html` 读取同一份文件，中文正常显示

---

### 二、排查思路

#### 第一步：找"能跑"与"不能跑"的边界

用户描述"ocr-local-test.html 能显示，Review 页面不行"。两者用的是同一份 PDF 源文件，差别只在**数据来源**：

| 场景 | 数据来源 |
|------|---------|
| ocr-local-test.html | 直接读 `ccr_result/` 下的本地 JSON 文件 |
| Review 页面 | 调用 `GET /documents/{id}/result` → 数据库 `result_json` |

**结论：优先怀疑数据源，而非渲染代码。**

#### 第二步：对比两个数据源的格式

直接查数据库里存的内容：

```python
# ccr_result/ 本地文件
{ "file name": "...", "number of pages": 75, "kids": [...1147个] }

# 数据库 result_json
{ "status": "success", "document": {"json_content": {...}}, "audit_metadata": {...} }
```

两者格式完全不同：
- 本地文件是 **kids 格式**（OpenDataLoader 旧格式）
- 数据库里是 **Docling 格式**（OpenDataLoader HTTP API 响应格式）

查数据库里的 `json_content`：`texts = 0, tables = 0, pictures = 0`，完全是空的。

**结论：数据库里存的是一份没有内容的 Docling 格式，渲染当然空白。**

#### 第三步：追溯写入数据库的代码，找写错的地方

按数据流反向追查：

```
数据库 ← RecognitionResult.result_json
         ← Celery task: process_document_task()
             ← OpenDataLoaderWrapper.convert()
                 ← 优先读本地 kids JSON（_read_old_format_result）
                 ← fallback: HTTP API 响应（Docling 格式）
```

重点看 `_read_old_format_result()` 为什么没有读到 kids JSON：

```python
# 原始代码：按文件 MD5 hash 找目录
file_hash = self._get_file_hash(file_path)
possible_paths = [
    f"/tmp/ocr_project/{file_hash}",  # ← 按 hash 命名的目录根本不存在
    ...
]
```

实际检查发现：OCR 引擎把 kids JSON 存在 `/tmp/ocr_project/{引擎自己的UUID}/` 下，这个 UUID 跟文档服务的 UUID 和文件 hash 毫无关系。所以查找失败，每次都 fallback 到空的 Docling 格式。

**根本原因：`_read_old_format_result()` 查找路径逻辑与 OCR 引擎实际保存路径不匹配。**

#### 第四步：实施修复

修改查找逻辑：不按 hash 目录找，改为扫描全部子目录，按**文件名 + 时间戳**（OCR 调用开始之后写入的最新文件）匹配：

```python
# 修复后
call_start = time.time()
response = requests.post(...)  # 先调用 OCR

# 扫描所有子目录，找调用之后写入的、名字匹配的 kids JSON
for entry in os.scandir("/tmp/ocr_project"):
    json_path = os.path.join(entry.path, f"{base_name}.json")
    if os.path.exists(json_path) and os.path.getmtime(json_path) >= call_start:
        data = json.load(open(json_path))
        if 'kids' in data:
            return data  # 命中
```

同时修复存量数据：将数据库里的 Docling 格式记录替换为从本地已有的 kids JSON 文件加载、经 audit_enhancer 增强后的版本。

---

### 三、为什么容易走弯路

直觉上看到"中文不显示"会先怀疑：
- 字体问题（Arial 不支持中文）
- CSS overflow 裁切
- 编码问题

这些都是**渲染层假设**。但正确的排查顺序应该是**先验证数据是否到位**，再看渲染。

本例中用一段 Python 脚本直接查数据库，发现 texts = 0，问题立刻清晰：不是渲染问题，是数据问题。

---

### 四、通用原则

1. **先验证数据，再看渲染**：界面空白时，先打印/查询原始数据，确认数据到没到位，再分析渲染代码。
2. **反向追溯数据流**：从"数据存在哪"反向找"谁写进去的"，逐层检查每一步是否正确执行。
3. **对比"能跑"和"不能跑"的数据源**：找到两者的格式差异，差异所在就是问题所在。

## 问题三：文档处理状态长期停留在"处理中"

**日期：** 2026-04-17
**涉及文件：** `backend/ocr/wrapper.py`、`backend/tasks/document_tasks.py`

---

### 一、问题现象

- 上传新文件后状态一直是"处理中"，不变成"已完成"也不报错
- 没有任何错误日志

---

### 二、排查思路

#### 第一步：查数据库，确认任务信息

```python
# 查到最新文档：status=PROCESSING，task_id=c73cb04a，error_message=None
```

有 task_id，没有错误，说明任务被正常分配给了 Celery worker，但没有执行完。

#### 第二步：查 Redis，看任务在 Celery 侧的状态

```python
r.get('celery-task-meta-c73cb04a...')
# → status: STARTED, pid: 99498
```

任务状态是 STARTED，说明 worker 已经**开始执行**，但一直没有结束（没有变成 SUCCESS 或 FAILURE）。

#### 第三步：确认 worker 进程是否还活着

```bash
ps -p 99498
# → 进程存在，已运行 5小时42分钟，CPU 占用 0%
```

进程活着但 CPU 为 0，说明它在**等待某个 IO**，没有在计算。

#### 第四步：找 IO 等待点

worker 执行的是 `process_document_task`，逐行看，唯一可能长时间阻塞的只有：

```python
response = requests.post(
    "http://localhost:5002/v1/convert/file",
    timeout=3600  # ← 1小时超时
)
```

检查 OCR Server：

```bash
curl http://localhost:5002/
# → 无响应（连接拒绝）
```

**根本原因：OCR Server 没有运行，worker 卡在 HTTP 请求等待，超时时间长达 1 小时，所以状态长期是"处理中"。**

---

### 三、为什么容易走弯路

没有错误日志，容易误判为：
- 数据库死锁
- Celery 队列堵塞
- Redis 连接问题

正确顺序：**先看任务在哪个阶段卡住**（Redis 里 STARTED），再找卡住点（进程 CPU=0 + 看代码 IO 等待点），最后验证依赖服务（OCR Server）是否正常。

---

### 四、通用原则

1. **分层定位"卡在哪一层"**：任务系统通常分为：调度层（任务是否入队）→ 执行层（worker 是否取到）→ 依赖层（依赖服务是否正常）。每层各有检查方法。
2. **CPU=0 且进程活着 = IO 等待**：这是找阻塞点的关键信号，立刻检查该进程在等什么外部资源。
3. **`timeout` 不能设太长**：生产代码里长时间阻塞的 IO 调用应设合理超时并捕获异常，否则无声地卡住比报错更难排查。

## 综合方法论：如何高效定位和解决 Bug

> 以下方法论从上面三个案例中提炼，适用于绝大多数前后端 Bug 排查场景。

---

### 一、定位阶段：找到"谁的问题"

**核心动作：找最小差异**

| 问法 | 意义 |
|------|------|
| "同样的功能，哪个场景能跑，哪个不能跑？" | 缩小问题范围 |
| "两个场景之间，唯一的差别是什么？" | 锁定嫌疑层 |
| "是数据的问题还是渲染/逻辑的问题？" | 分层排除 |

**案例对应：**
- PDF 渲染失败：`pdf-test.html` 能跑、Vue 组件不能跑 → 差别在 Vue 响应式层
- OCR 渲染空白：`ocr-local-test.html` 能跑、Review 页面不行 → 差别在数据来源（本地文件 vs 数据库）
- 状态卡"处理中"：有 task_id、无错误 → 差别在执行层（worker 卡住而非未启动）

---

### 二、诊断阶段：相信第一手证据

**原则：直接查，不要猜**

| 现象 | 正确动作 | 错误动作 |
|------|---------|---------|
| 界面空白 | 先打印原始数据，确认数据是否存在 | 直接改渲染代码 |
| 报错信息 | 逐字解读，找出它指向的具体对象/操作 | 搜索报错全文，套用他人解决方案 |
| 任务卡住 | 查任务在哪一层（队列/执行/依赖），找 IO 等待点 | 重启服务、清队列等无目标操作 |

**数据流反向追溯法：**

```
界面现象
  ↓ 前端拿到的数据是什么？（打印 API 响应）
API 返回
  ↓ API 从哪里取的数据？（看 service 层）
Service 层
  ↓ Service 读的是数据库还是缓存？（直接查数据库）
数据库 / Redis
  ↓ 谁写进去的？写入逻辑是否正确？（反查写入代码）
写入代码（根本原因）
```

从现象出发，沿数据流反向追查，在哪一层数据开始"不对"，问题就在那一层。

---

### 三、修复阶段：最小改动，有迹可查

**原则：只改必须改的，改完立刻验证**

1. **改之前说清楚假设**：`_read_old_format_result` 的查找路径假设"引擎按 hash 命名目录"，但实际是"引擎自己生成 UUID"，假设错误导致每次 fallback。
2. **改之后立刻验证**：修改 wrapper.py 后，用 Python 脚本直接调用新逻辑，确认能找到正确文件，再考虑存量数据修复。
3. **分离"修旧数据"和"修代码逻辑"**：代码逻辑修复保证新数据正确，存量数据用一次性脚本修复，两者不要混在一起。

---

### 四、防御阶段：让问题不再复现

每次修完 Bug，问自己：

| 问题 | 本例答案 |
|------|---------|
| 为什么当时没被发现？ | 测试用的是本地文件，没有走数据库链路 |
| 下次怎么更快发现？ | 数据写入时做格式断言，检查 `kids` 字段是否存在 |
| 类似的地方还有吗？ | `convert_chunk` 也有同样逻辑，一并检查 |

---

### 五、快速排查检查清单

遇到 Bug 时，按顺序过这张清单：

```
□ 1. 找边界：哪个场景能跑，哪个不行？差别是什么？
□ 2. 分层：是数据问题、逻辑问题，还是依赖问题？
□ 3. 查第一手数据：直接查数据库/日志/Redis，不要只看界面
□ 4. 读报错/状态：逐字解读，不要搜索关键词套方案
□ 5. 反向追溯：沿数据流从现象往源头走，找第一个"不对"的地方
□ 6. 最小修改：只改导致问题的那一行/那一处，不顺手重构
□ 7. 立刻验证：改完后用脚本/curl/日志直接确认，不要只靠界面
□ 8. 举一反三：同样的假设在其他地方有没有用到？
```
