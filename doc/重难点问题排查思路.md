# 重难点问题排查思路分析

> 本文档记录开发过程中遇到的重难点问题，重点梳理排查思路，供团队参考。

---

## 问题一：PDF 渲染失败 —— Vue 3 响应式系统与 pdfjs 私有字段冲突

**日期：** 2026-04-16  
**耗时：** 同事排查一上午未解决  
**涉及文件：** `frontend/src/components/compare/PDFViewer.vue`

---

### 一、问题现象

- 访问 `http://localhost:3000/pdf-test.html` → PDF 正常渲染
- 访问前端核验页面，左侧源文件区域 → 渲染失败，报错：

```
PDF加载失败: Cannot read private member #pagePromises from an object whose class did not declare it
```

---

### 二、排查思路

#### 第一步：找到"能跑"与"不能跑"的边界

两个场景唯一的差异：

| 场景 | 运行环境 |
|------|---------|
| `pdf-test.html` 正常 | 浏览器原生 `<script type="module">`，不经过 Vite |
| 前端页面失败 | Vue 组件，经过 Vite 打包 + Vue 响应式系统 |

**结论：问题出在 Vite 或 Vue 这一层，与 pdfjs 本身无关。**

#### 第二步：精读报错信息

```
Cannot read private member #pagePromises from an object whose class did not declare it
```

这句话的精确含义：**访问 `#pagePromises` 时，`this` 不是声明这个私有字段的那个类的实例。**

JS 私有字段的规则：只有通过**原始类实例**才能访问，任何包装（Proxy、跨模块实例）都会触发此错误。

由此得出两个排查方向：
1. **多实例**：`getDocument()` 和 `getPage()` 来自两个不同的 pdfjs 模块副本
2. **对象被包装**：pdfjs 返回的对象被某个东西套了一层，`this` 变了

#### 第三步：用堆栈信息排除多实例

查看完整报错堆栈，两处 `getPage` 都指向同一个 `/pdf.mjs` 文件的同一行，说明**不存在多实例**，方向1排除。

#### 第四步：锁定"谁包装了对象"

排除多实例后，唯一可能就是**对象在 Vue 内部被包装**。定位到代码：

```js
const pdfDoc = ref(null)  // ← 嫌疑点
```

Vue 3 的 `ref()` 在存入对象时会调用内部的 `reactive()`，对对象做**深度 Proxy 包装**。此后 `pdfDoc.value` 拿到的不是原始的 `PDFDocumentProxy` 实例，而是它的 Proxy 代理。

当 pdfjs 内部执行 `this.#pagePromises` 时，`this` 是 Proxy，不是原始类实例，JS 引擎拒绝访问私有字段，抛出上述错误。

#### 第五步：验证假设，实施修复

假设成立，解决方案明确：**阻止 Vue 对 pdfjs 对象做深度代理。**

Vue 提供了两种工具：
- `shallowRef()`：只追踪 `.value` 引用本身，不深度代理内部属性
- `markRaw()`：标记对象永远不被 Vue 代理

**修复方式：**

```js
// ❌ 修复前 —— ref() 会深度代理，破坏私有字段访问
const pdfDoc = ref(null)
const pages = ref([])

// ✅ 修复后 —— shallowRef() 只追踪引用，不代理内部对象
const pdfDoc = shallowRef(null)
const pages = shallowRef([])
```

**附带注意事项：** `shallowRef` 包裹的数组，`.push()` 不会触发视图更新，必须重新赋值：

```js
// ❌ 不触发更新
pages.value.push({ num: i, page })

// ✅ 正确写法
pages.value = [...pages.value, { num: i, page }]
```

---

### 三、为什么容易走弯路

同事排查方向集中在：pdfjs 版本兼容性、Vite 配置（optimizeDeps）、worker 加载路径等，这些都是**表面现象**，本质上没有抓住"是谁改变了 `this`"这个核心。

常见无效方案：

| 尝试方案 | 为何无效 |
|---------|---------|
| 改用 `legacy/build/pdf.mjs` | legacy 是 CJS 格式，Vite 处理时问题依旧 |
| 调整 `optimizeDeps.include/exclude` | 不是 Vite 预构建问题 |
| 用 `import.meta.url` 指定 workerSrc | 不是 worker 多实例问题 |
| 用动态 `import()` 加载 public 目录 | Vite 明确禁止 import public 目录的 JS 文件 |

---

### 四、通用排查原则（可复用）

1. **先找边界**：找到"能跑"和"不能跑"的最小差异，问题范围立即缩小一半。
2. **相信报错信息**：报错说什么就是什么，不要绕开它，要正面解释它。本例中"class did not declare it"直接指向 `this` 被替换，顺着这条线找到 Vue Proxy 即可。
3. **第三方库对象存入 Vue 响应式时，优先用 `shallowRef` / `markRaw`**：凡是含有私有字段、内部状态、原生 API 对象（canvas、WebSocket、pdfjs、mapbox 等）都不应该被 Vue 深度代理。
